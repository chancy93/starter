# 启步器 · 产品思考文档

> 这不是技术文档。这份文档回答一个问题：**为什么做这个产品，以及我怎么知道它做对了。**

---

## 一、我观察到了什么问题？

身边同学（包括我自己）在面对大任务时，最常见的卡点不是"不会做"，而是**"启动不了"**。

典型场景：
- 写论文：打开 Word → 发呆 → 刷手机 → 一天过去了
- 整理房间：想着"等周末再说"→ 三个周末过去了
- 健身：运动服都买了 → "今天太累了明天开始"

这不是懒，是**任务颗粒度太粗导致的心理抗拒**。当一个任务在你脑子里是"写论文"三个字时，它是一团模糊的巨大阻力。但如果你把它变成"打开文档 → 写标题 → 写第一段草稿"，每一步都变得具体可执行。

**核心洞察：降低启动门槛的关键，不是提高意志力，而是降低每一步的认知负担。**

## 二、目标用户

### 主要用户画像
| 维度 | 描述 |
|------|------|
| 人群 | 18-28 岁学生 / 初入职场人群 |
| 特征 | 有明确任务目标，但容易拖延，自责"自制力差" |
| 场景 | 写论文、备考复习、整理房间、健身启动 |
| 心理 | 知道该做什么但"启动困难"，需要外部推一把 |

### 非目标用户
- 已经有成熟 GTD 工作流的职场人（Todoist 更适合他们）
- 需要团队协作的项目管理者（这是个人工具）

## 三、核心假设

### 假设 1：微步骤拆解假说
> "如果用户看到的是 3-5 个可立即执行的微步骤，而不是一个模糊的大任务，启动概率会显著提高。"

**验证方式**：统计子任务"启动"按钮的点击率。如果 AI 拆解后的任务点击率 < 50%，说明拆解力度不够或用户不信任拆解结果。

### 假设 2：5 秒倒计时假说
> "倒计时提供了'准备就绪'的心理缓冲，比直接开始更能降低启动的心理阻力。"

**默认 5 秒的理由**：太短（3 秒以下）没缓冲效果，太长（10 秒以上）用户可能走神。5 秒是"现在我必须开始了，但还有一口气"的临界点。

**验证方式**：对比不同倒计时设置下，用户完成第一步的概率。

### 假设 3：拍照打卡强化承诺
> "拍照行为将'我要做 X'具象化为一个物理动作，增强完成承诺。"

**验证方式**：对比"跳过拍照"和"拍照打卡"两组的完成率差异。

### 假设 4：全流程闭环 > 单步骤提醒
> "从拆解到完成到成就记录，完整闭环比单独提醒更能建立行为惯性。"

## 四、竞品分析

| 产品 | 核心机制 | 解决了什么 | 没解决什么 |
|------|---------|-----------|-----------|
| **Forest** | 专注计时 + 种树 | 过程专注 | 不帮你拆解任务，不知道从哪开始 |
| **Todoist** | 清单管理 | 任务组织 | 清单本身不驱动行动，缺少"启动推力" |
| **Microsoft To Do** | 每日计划 | 日程规划 | 任务颗粒度由用户自己定义，容易写得太粗 |
| **Fabulous** | 习惯养成 | 行为惯性 | 偏向日常习惯，不适合"一次性大任务" |
| **ChatGPT** | 对话拆解 | 通用智能 | 拆完就拆完了，没有执行闭环 |

**启步器的定位**：在"拆解"和"执行"之间做桥梁。不是帮你管理所有任务，而是**专门帮你把"最难开始的那个任务"开个头**。

这个定位意味着我们不和 Todoist 比组织能力，不和 Forest 比专注计时——我们解决的是**T-1 时刻的启动阻力**。

## 五、用户旅程

```
输入大任务 → AI 拆解为 3-5 个微步骤
                ↓
         选一个步骤点"启动"
                ↓
         5 秒倒计时（心理缓冲）
                ↓
         拍照打卡（行为承诺）
                ↓
         确认完成 ✅
                ↓
         能量进度条涨一格
                ↓
         全部完成 → 🔥 庆祝动画 → 写入成就墙
```

**关键设计决策**：
1. **拆解在输入时完成，不在执行中**：用户只需要"选和做"，不需要"想"
2. **每步独立启动**：不用一次性承诺全部做完，降低心理负担
3. **拍照非强制**：但如果拍了，完成确认感更强
4. **成就墙持久化**：让用户看到"我已经做完了 12 个大任务"，建立正向反馈循环

## 六、北极星指标

**主指标：周任务启动率**
= 本周点击"启动"的次数 / 本周打开应用的次数

这个指标衡量的是：用户看到任务后，**有没有真的按下启动按钮**。

### 辅助指标
| 指标 | 含义 |
|------|------|
| 子任务完成率 | 启动后是否走到了 ✅ 确认完成 |
| 全任务通关率 | 所有子任务都完成的任务占比 |
| 7 日留存 | 一周后是否还回来用 |
| AI 拆解满意度 | 用户是否修改了 AI 的拆解结果（修改 = 不满意） |

## 七、迭代路线图

### V1.0 — 当前版本 ✅
- [x] AI 任务拆解（DeepSeek API）
- [x] 倒计时启动 + 拍照打卡
- [x] 能量进度条 + 成就墙
- [x] PWA 离线支持
- [x] localStorage 持久化

### V1.1 — 数据驱动（下一步）
- [ ] 接入埋点（Umami / GA4），收集 5 个关键事件
- [ ] 找 5-10 个真实用户试用一周
- [ ] 收集用户访谈反馈，验证核心假设
- [ ] 修复 API Key 前端暴露问题

### V1.2 — AI 能力升级
- [ ] 根据任务类型自适应调整拆解粒度
- [ ] 基于用户历史完成率推荐最优倒计时秒数
- [ ] 拆解结果不满意时支持"重新拆解"并收集反馈
- [ ] 输出更有鼓励性的拆解文案（而非机械的步骤列表）

### V2.0 — 行为闭环
- [ ] 每周回顾："你这周启动了 X 次，完成了 Y 个任务"
- [ ] 连续打卡天数徽章（行为激励机制）
- [ ] 允许用户自定义"我的常用任务模板"（减少重复输入）
- [ ] 分享成就卡片到社交媒体

### 不做的事
- 不做团队协作（保持个人工具定位）
- 不做日历集成（避免功能膨胀）
- 不做复杂的时间统计（不是复盘工具，是启动工具）

## 八、已知风险 & 思考

| 风险 | 影响 | 缓解方案 |
|------|------|---------|
| AI 拆解结果不合用户预期 | 用户不再信任拆解功能 | 允许一键重新拆解 + 手动编辑 |
| 用户新鲜感过后流失 | 留存下降 | 成就墙 + 每周回顾 + 习惯绑定 |
| DeepSeek API 不稳定 | 拆解功能不可用 | 已有 fallback 机制，后续可加模型热备 |
| 手机端网页体验不如原生 App | 用户不愿添加到主屏幕 | PWA 已实现，后续可考虑套壳上架 |
| 竞争壁垒低 | 大厂可快速复制 | 专注细分场景（启动困难）+ 数据飞轮 |
